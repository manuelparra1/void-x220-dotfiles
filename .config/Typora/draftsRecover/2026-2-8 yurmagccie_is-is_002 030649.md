Every IS-IS router belongs to an Area. The idea of this comes from the fact that IS-IS was initially created to route Connectionless Network Protocol (CLNP) where the address belongs to a device (Router), whereas in Internet Protocol (IP) the address belongs to the particular interface.

IS-IS protocol has two levels or hierarchy:  
– Level-2 (**backbone**)  
– Level-1 (**non-backbone**)

A **backbone area** in IS-IS is a **contiguous** collection of Level-2 routers.

Every non-backbone area can reach other non-backbone area **only through thr backbone area**.

There are three IS-IS router types:  
– Level-1 (L1)  
– Level-2 (L2)  
– Level-1-2 (L1-2)

**Level-1 router**  
Level-1 router is the router that is configured to form Level-1 adjacencies only. Level-1 adjacencies can be built between routers that reside **in the same area**.

**LSP**s within L1 adjacencies contain **only connected prefixes**.

**Level-2 router**  
Level-2 router is the router that is configured to form Level-2 adjacencies only. Level-2 adjacencies can be built between IS-IS routers that configured with **any area number**.

The Level-2 LSDB has all routing information about **intra-area** and **inter-area** prefixes

**Level-1-2 router**  
Level-1-2 router is the router that is configured to form both Level-1 and Level-2 adjacencies. Such router is called Area Border Router (ABR).  
Level-1-2 router has two separate LSDBs:  
– Level-1 LSDB  
– Level-2 LSDB

LSPs within L1 adjacencies contain only connected prefixes.  
LSPs within L2 adjacencies contain connected prefixes and prefixes, that were **implicitly imported** from L1 LSDB.

![IS-IS2.draw_io-IS-IS2.1.L1-2](./images/is-is2-draw_io-is-is2-1-l1-2.jpg)
As you can see, there is no reason to have both L1 and L2 adjacencies over a single link, except one case that will be discussed next.

**Area Border Router (ABR)**  
In IS-IS, the **backbone area** is represented by a **set of a router**s which form **L2 adjacency** with each other. In other words, **Area-ID is irrelevant for the backbone area**.

**Area-ID is** **significant for a non-backbone areas**. Due to this fact ABR must belong to a non-backbone area.

![IS-IS2.draw_io-IS-IS2.2.Areas](./images/is-is2-draw_io-is-is2-2-areas.jpg)
An ABR advertises its L1-LSPs to its L1 neighbors with **Attachment Bit** (ATT-bit) set:

```
R1#sh isis database l1

Tag 1:
IS-IS Level-1 Link State Database:
LSPID                 LSP Seq Num  LSP Checksum  LSP Holdtime/Rcvd      ATT/P/OL
R1.00-00            * 0x0000000F   0xF03F                1031/*         1/0/0
R2.00-00              0x0000000F   0x46DC                 469/1198      1/0/0
R6.00-00              0x00000010   0x8594                1032/1198      0/0/0
R7.00-00              0x0000000D   0xC85A                 444/1199      0/0/0
```

This results in **the default route** towards to the router with ATT-bit. This default route is installed by other routers in the area:

```
R6#sh ip route

! omitted !

Gateway of last resort is 10.1.26.2 to network 0.0.0.0

i*L1  0.0.0.0/0 [115/10] via 10.1.26.2, 01:24:00, Ethernet0/0
```

**Link-state protocol route preference**  
IS-IS prefers L1 routes (inter-area) to L2 routes (intra-area). This rule may cause sub-optimal routing. R2 will forward packet out of the interface **Eth0/2 via R6/R7** to get to the network in area 67 that is connected to R1, **instead** of going **directly** via the interface **Eth0/1**. R4 can go directly to R3, since link between them is in **both L1 and L2** LSDBs. Check out traceroute results:

![IS-IS2.draw_io-IS-IS2.3.Suboptimal.jpg](./images/is-is2-draw_io-is-is2-3-suboptimal1.jpg)
In conclusion, the link between ABRs should be included in both topologies – L1 and L2.

**ABR Prefix Leaking**  
By default, **no prefixes from L2 LSDB (backbone) are imported to L1 LSDB**, but there is a way to do that.

[[RFC5305](https://tools.ietf.org/html/rfc5305#section-4.1)]: To prevent this looping of prefixes between levels, the up/down bit logic is used. The up/down bit SHALL be **set to 0** when a prefix is **first injected** into IS-IS. If a prefix is advertised from a higher level to a lower level (e.g., level 2 to level 1), the bit MUST be **set to 1**, indicating that the prefix has **traveled down** the hierarchy. Prefixes that have the up/down bit set to 1 may only be advertised down the hierarchy, i.e., to lower levels.

Here is an example, how to import only loopback prefixes from backbone in L1 topology:

```
router isis 1
 redistribute isis ip level-2 into level-1 route-map rm-isis
!
ip prefix-list pl-loopback seq 5 permit 0.0.0.0/0 ge 32
!
route-map rm-isis permit 10
 match ip address prefix-list pl-loopback
```

As a result:

```
R1#sh isis database l1 R1.00-00 detail

Tag 1:

IS-IS Level-1 LSP R1.00-00
LSPID                 LSP Seq Num  LSP Checksum  LSP Holdtime/Rcvd      ATT/P/OL
R1.00-00            * 0x00000009   0x89D3                1169/*         1/0/0
  Area Address: 67
  NLPID:        0xCC
  Hostname: R1
  Metric: 10         IS R7.00
  Metric: 10         IS R2.00
  IP Address:   192.168.0.1
  Metric: 10         IP 10.0.1.0 255.255.255.0
  Metric: 10         IP 10.1.17.0 255.255.255.0
  Metric: 10         IP 192.168.0.1 255.255.255.255
  Metric: 10         IP 10.1.12.0 255.255.255.0
  Metric: 30         IP-Interarea 192.168.0.3 255.255.255.255
  Metric: 30         IP-Interarea 192.168.0.4 255.255.255.255
  Metric: 20         IP-Interarea 192.168.0.5 255.255.255.255
  Metric: 40         IP-Interarea 192.168.0.8 255.255.255.255
  Metric: 40         IP-Interarea 192.168.0.9 255.255.255.255
```

**IP-Interarea** indicates that up/down bit is set.

**Summarization**  
Prefix summarization can be done on ABR. Summarization can be applied only to prefixes that **are leaked between LSDBs**. For example, if you want to have summarization route **in L2 LSDB** – no problem here because L1 prefixes are imported into L2 LSDB by default. If you want to have summary route **in L1 LSDB**, you need to **explicitly import prefixes from L2** (leaking), because, by default, L2 prefixes are no imported into L1.

**Case 1. L1 to L2 summarization.**  
R6’s and R7’s Loopbacks can be summarized:

```
R1#sh ip ro 192.168.0.6 255.255.255.254 longer-prefixes | i L1
       i - IS-IS, su - IS-IS summary, L1 - IS-IS level-1, L2 - IS-IS level-2
i L1     192.168.0.6/32 [115/30] via 10.1.17.7, 01:47:23, Ethernet0/2
i L1     192.168.0.7/32 [115/20] via 10.1.17.7, 01:49:03, Ethernet0/2
```

Configuration for this will be applied on R1:

```
router isis 1
 summary-address 192.168.0.6 255.255.255.254
```

When you summarize prefixes, the route towards to Null 0 interface is created by default:

```
R1#sh ip ro 192.168.0.6 255.255.255.254 longer-prefixes | i i

Gateway of last resort is not set
      192.168.0.0/24 is variably subnetted, 10 subnets, 2 masks
i su     192.168.0.6/31 [115/20], 00:10:27, Null0
i L1     192.168.0.6/32 [115/30] via 10.1.17.7, 01:49:02, Ethernet0/2
                        [115/30] via 10.1.12.2, 01:49:02, Ethernet0/1
i L1     192.168.0.7/32 [115/20] via 10.1.17.7, 01:50:42, Ethernet0/2
```

More specific routes (192.168.0.6/32 and 192.168.0.7/32) will be excluded from L2 LSDB:

```
R1#sh isis database l2 de R1.00-00 | i 192
  IP Address:   192.168.0.1
  Metric: 10         IP 192.168.0.1 255.255.255.255
  Metric: 20         IP 192.168.0.2 255.255.255.255
  Metric: 20         IP 192.168.0.6 255.255.255.254
```

While R2 still has these:

```
R1#sh isis database l2 de R2.00-00 | i 192
  IP Address:   192.168.0.2
  Metric: 20         IP 192.168.0.1 255.255.255.255
  Metric: 10         IP 192.168.0.2 255.255.255.255
  Metric: 20         IP 192.168.0.6 255.255.255.255
  Metric: 30         IP 192.168.0.7 255.255.255.255
```

**Case 2. L2 to L1 summarization.**  
I cannot think of a case why would you summarize prefixes from L2 to L1, since there is the ultimate summarization by default – thanks to ATT-bit.

Summarization is applied to leaked prefixes. Since, by default, no prefixes are leaked from L2 to L1, we need to configure one, for example, on R2:

```
router isis 1
 redistribute isis ip level-2 into level-1 route-map rm-isis
!
ip prefix-list pl-loopback seq 5 permit 0.0.0.0/0 ge 32
!
route-map rm-isis permit 10
 match ip address prefix-list pl-loopback
```

Now, summary route can be put into L1 LSDB:

```
router isis 1
 summary-address 192.168.0.0 255.255.255.0 level-1
```

Here is the summary route:

```
R2#sh isis database l1 detail R2.00-00 | i 192
  IP Address:   192.168.0.2
  Metric: 10         IP 192.168.0.2 255.255.255.255
  Metric: 20         IP-Interarea 192.168.0.0 255.255.255.0
```

**Filtering from L1 to L2**  
Prefixes from L1 can also be filtered and not imported into L2 LSDB by configuring explicit redistribution from L1 LSDB to L2 LSDB. Configuration to apply on R2:

```
router isis 1
 redistribute isis ip level-1 into level-2 route-map rm-isis
!
ip prefix-list pl-loopback seq 5 permit 0.0.0.0/0 ge 32
!
route-map rm-isis permit 10
 match ip address prefix-list pl-loopback
```

And now R2 imports from L1 only /32 prefixes from L1:

```
R2#sh isis database l2 detail R2.00-00
```
